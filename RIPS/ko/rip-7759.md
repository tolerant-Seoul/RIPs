## 개요

이 문서는 레이어 2 트랜잭션 수수료에 대한 최소한의 비즈니스 및 기술 전제 조건, 기능 및 비기능 요구사항을 설명합니다. 이를 구현하면 레이어 2 솔루션이 트랜잭션 수수료를 투명하고 일관되게 계산하고 표시할 수 있습니다.

## 동기

레이어 2(L2) 트랜잭션 수수료는 합의 프로토콜의 검증자와 같이 경제적 보안 보장을 제공하는 참여자들에게 주어지는 보상 외에 L2 플랫폼 운영 자금을 조달하는 중요한 요소입니다. 그러나 이러한 트랜잭션 수수료가 어떻게 산출되는지, 누구에게 지급되는지, L2 플랫폼의 참여자들에게 어떻게 그리고 어디서 표시되는지는 L2 플랫폼마다 크게 다릅니다.

위의 사항과 개방된 생태계에서 신뢰를 구축하기 위한 투명성의 필요성, 그리고 수수료 투명성과 부과할 수 있는 수수료 유형에 대한 진화하는 법적 및 규제 환경을 고려하여, 이 문서는 다양한 유형의 L2 트랜잭션 수수료를 정의하고 L2 플랫폼을 위한 트랜잭션 수수료 투명성에 대한 요구사항을 정의합니다.

따라서 오픈소스 이니셔티브인 OASIS가 관리하는 EEA 커뮤니티 프로젝트의 L2 WG는 이 문서가 L2 트랜잭션 수수료에 대한 요구사항을 명시하도록 의도합니다. 이 문서는 먼저:

- 트랜잭션 수수료와 그 구성 요소의 의미를 정의합니다
- 트랜잭션 수수료와 관련된 역할을 정의합니다
- 트랜잭션 유형과 트랜잭션 수수료와의 관계를 정의합니다

이후, 문서는 수수료 투명성, 수수료 추정, 최종 수수료, 그리고 수수료가 다양한 역할에 어떻게, 언제, 어디서 전달되는지에 대한 요구사항을 정의합니다. 사양은 또한 역할 간의 책임과 의무에 대한 요구사항을 논의합니다.

참고로, L2 플랫폼 간의 자산 및 데이터 브릿지와 L2 플랫폼과 중앙화 거래소 간의 자산 및 데이터 브릿지와 관련된 수수료는 이 문서의 범위를 벗어납니다.

## 사양

### 표기 규칙: 요구사항 ID

요구사항은 요구사항 수준과 요구사항 번호로 구성된 고유 ID로 식별됩니다. 규칙 **[RequirementLevelRequirementNumber]**에 따릅니다.
아래 규칙에 따라 요구사항 ID에 코딩된 네 가지 요구사항 수준이 있습니다:

**[R]** - ID가 문자 _R_로 시작하는 요구사항의 요구사항 수준은 [RFC2119](https://www.rfc-editor.org/rfc/rfc2119)에 설명된 대로 **MUST**로 해석됩니다. \
**[D]** - ID가 문자 _D_로 시작하는 요구사항의 요구사항 수준은 [RFC2119](https://www.rfc-editor.org/rfc/rfc2119)에 설명된 대로 **SHOULD**로 해석됩니다. \
**[O]** - ID가 문자 _O_로 시작하는 요구사항의 요구사항 수준은 [RFC2119](https://www.rfc-editor.org/rfc/rfc2119)에 설명된 대로 **MAY**로 해석됩니다.

요구사항은 각 요구사항 수준 내에서 오름차순으로 고유하게 번호가 매겨집니다.

예시: [R1]은 사양의 절대적 요구사항으로 읽어야 하며, [D1]은 권장사항이고 [O1]은 진정으로 선택사항입니다.

### L2 트랜잭션 수수료 및 관련 개념의 정의

이 문서는 트랜잭션 수수료를 레이어 2(L2) 네트워크 또는 프로토콜 토큰으로 트랜잭션 발신자 또는 트랜잭션 전송자가 지불해야 하는 수수료로 정의하며, 이는 실행 수수료, 데이터 수수료 및 우선순위 수수료의 합계로 구성됩니다.

이 문서는 트랜잭션을 L2 계정에서 전송된 디지털 서명 메시지로 정의하며, 이는 L2 상태 전이를 초래하는 명령과 데이터를 포함합니다. 또한 이 문서는 상태와 상태 전이를 다음과 같이 정의합니다:

* 상태: 암호화 데이터 구조를 사용하여 서로 매핑된 L2의 모든 계정 집합
* 상태 전이: L2의 전체 상태를 구성하는 모든 계정 집합에서 하나 이상의 계정을 결정적으로 변경하는 이벤트

또한 이 문서는 계정을 최소한 고유 계정 식별자, 논스라고도 하는 결정적 순서 매개변수, 레이어 1, 레이어 2 또는 사이드체인 교환 단위의 잔액(프로토콜 토큰이라고도 함)으로 구성된 단위로 정의합니다. 계정의 변경은 고유한 암호화 공개 및 개인 키 쌍에 의해 제어됩니다. 일반적으로 계정 변경을 결정하는 계정과 관련된 명령과 데이터를 참조하는 추가 계정 요소가 있습니다.

공식으로 트랜잭션 수수료는 다음과 같이 주어집니다:

**트랜잭션 수수료 = 실행 수수료 + 데이터 수수료 + 우선순위 수수료**

다양한 구성 요소는 다음과 같이 정의됩니다:

* **실행 수수료:** 우선순위 수수료와 데이터 수수료를 제외한 레이어 2 트랜잭션 수수료를 충당하기에 충분한 트랜잭션 발신자 또는 트랜잭션 전송자가 지불해야 하는 수수료

* **데이터 수수료:** 레이어 1 트랜잭션 수수료와 비레이어 1 데이터 저장소와 관련된 기타 트랜잭션 데이터 수수료를 충당하기에 충분한 트랜잭션 발신자 또는 트랜잭션 전송자가 지불해야 하는 수수료, 우선순위 수수료와 실행 수수료 제외

* **우선순위 수수료:** 트랜잭션 발신자 또는 트랜잭션 전송자가 새 블록에서 트랜잭션의 원하는 슬롯을 얻기 위해 지불합니다. 새 블록에서 트랜잭션의 정확한 위치인 슬롯은 우선순위 수수료 외에도 타임스탬프 또는 블록의 최대 추출 가능 가치(MEV) 최소화와 같은 요인에 의해 결정될 수 있습니다. MEV는 블록에 트랜잭션을 포함, 제외 및 순서 변경하여 표준 블록 보상과 가스 수수료를 초과하여 블록 생산에서 추출할 수 있는 최대 가치로 정의됩니다.

트랜잭션 수수료에는 이 문서의 맥락에서 L2 가스 가격 또는 L2 가스 등가 연산 및 저장소 소비 단위의 가격으로 정의되는 트랜잭션 수수료 가격이 있습니다.

가스 가격 또는 L2 가스 등가 연산 및 저장소 소비 단위의 가격은 일반적으로 가변적이며 L2 네트워크의 사용 수준에 따라 변합니다. 가스 가격은 EVM 호환 네트워크의 경우 기가 웨이(GWEI)로 표시되며, 웨이는 가스의 가장 작은 단위를 나타냅니다; 이더리움의 네이티브 토큰에서 1 웨이 = 10<sup>-18</sup> Eth입니다.

또한, 이 문서에서 가스는 EVM 호환 네트워크에서 특정 작업을 실행하는 데 필요한 연산 및 저장소 노력의 양을 측정하는 단위를 말합니다.

### 트랜잭션 수수료와 관련된 L2 역할의 정의

다음 역할은 트랜잭션 수수료, 그 계산 및 표시 방법, 시기, 위치와 관련이 있습니다:

* **트랜잭션 발신자:** 레이어 1, L2 또는 사이드체인을 위한 트랜잭션을 생성한 계정
* **트랜잭션 전송자:** 레이어 1, L2 또는 사이드체인에 트랜잭션을 보낸 계정
* **중개자:** 트랜잭션 발신자가 아닌 트랜잭션의 전송자인 엔티티
* **시퀀서:** 트랜잭션을 수집하고, L2가 운영되는 레이어 1에 배치로 게시하며, 게시된 트랜잭션에서 트랜잭션 수수료를 받고, 다른 L2 프로토콜 참여자에게 L2 수수료를 지불하며, 필요한 경우 다른 시퀀서와 합의 알고리즘에 참여하여 블록에서 트랜잭션 순서를 결정합니다.
* **L2 운영자:** L2 운영을 담당하는 엔티티
* **개발자:** 레이어 1, L2 또는 사이드체인에서 작동하는 소프트웨어 애플리케이션을 위한 컴퓨터 코드를 작성하는 개인

### 트랜잭션 유형의 정의

다양한 유형의 트랜잭션이 있으며, 상황에 따라 트랜잭션 수수료는 다른 역할에 의해 다르게 지불됩니다.

이 문서의 맥락에서 두 가지 유형의 트랜잭션이 있습니다:

* **직접 트랜잭션:** 트랜잭션 발신자가 트랜잭션 전송자이기도 한 트랜잭션
* **메타 트랜잭션:** 트랜잭션 전송자가 트랜잭션 발신자가 아니며, 동일한 트랜잭션이 직접 트랜잭션인 경우 트랜잭션 발신자의 트랜잭션 수수료가 다른 트랜잭션

직접 트랜잭션의 경우 트랜잭션 발신자는 트랜잭션 발신자가 최대로 지불할 의향이 있는 트랜잭션 수수료 가격에 따라 트랜잭션 수수료를 지불합니다.

메타 트랜잭션의 경우, 트랜잭션 발신자는 트랜잭션 수수료의 일부만 지불하거나 전혀 지불하지 않을 수 있습니다. 대신, 중개자가 트랜잭션 수수료의 전부 또는 일부를 지불합니다. 중개자는 트랜잭션 전송자이거나 발생한 트랜잭션 수수료에 대해 트랜잭션 전송자에게 지불할 수 있습니다. 또한 트랜잭션 발신자는 중개자 또는 트랜잭션 전송자 또는 둘 다에게 트랜잭션 수수료를 선불하거나 중개자에게 전송된 트랜잭션과 함께 지불을 제출할 수 있습니다.

따라서 트랜잭션 수수료와 그 구성 요소의 전달은 다양한 비즈니스 시나리오에 따라 다릅니다. 이 문서는 비즈니스 시나리오별이 아니라 역할별로 요구사항을 분류하고 해당 역할이 트랜잭션 수수료의 일부 또는 전부를 지불할 책임이 있는지에 따라 분류합니다.

### 레이어 2 트랜잭션 수수료 투명성 요구사항

<a name="r1"> **[R1]** </a>
L2 트랜잭션 수수료는 실행 수수료, 데이터 수수료 및 우선순위 수수료의 합계로 구성되어야 합니다(MUST).

[[R1]](#r1) 테스트 가능성: 수학적 등식은 등식 요소의 합이 원하는 출력과 같은지 테스트할 수 있습니다. 입력으로서 요소의 합이 원하는 출력과 같으면 테스트가 통과됩니다. 등식 요소의 합이 원하는 출력과 같지 않으면 테스트가 실패합니다.

<a name="r2"> **[R2]** </a>
L2 트랜잭션 수수료의 모든 구성 요소는 0보다 작아서는 안 됩니다(MUST NOT).

[[R2]](#r2) 테스트 가능성: 수학적 등식 또는 부등식은 테스트 가능한 조건으로 표현될 수 있습니다. 통과하는 테스트 조건은 트랜잭션 수수료의 각 개별 요소가 0 이상인 것입니다. 실패하는 테스트 조건은 하나 이상의 트랜잭션 수수료 요소가 0보다 작은 것입니다.

<a name="o1"> **[O1]** </a>
L2 트랜잭션 수수료 또는 그 구성 요소는 0일 수 있습니다(MAY BE).

[[O1]](#o1) 테스트 가능성: 수학적 등식은 테스트 가능한 조건으로 표현될 수 있습니다. [[R2]](#r2)에 대한 테스트는 [[R2]](#r2)의 테스트에서 0 값이 허용되므로 이 선택적 요구사항도 테스트합니다.

<a name="r3"> **[R3]** </a>
실행 수수료의 설정 및/또는 계산은 잘 문서화되고 검증 가능해야 합니다(MUST).

[[R3]](#r3) 테스트 가능성:

전제 조건:

* L2에 정의된 실행 수수료가 있어야 합니다.
* 현재 실행 수수료에 대한 문서가 있어야 합니다.
* L2 인스턴스가 실행 중이어야 합니다.

테스트 단계:

1. 실행 수수료 설정 및/또는 계산에 대한 문서화된 프로세스가 있는지 확인합니다.
2. 문서에 실행 수수료를 결정하는 데 사용되는 값, 특정 공식 또는 계산이 포함되어 있는지 확인합니다.
3. 문서에 계산에 사용되는 관련 가정 또는 종속성이 포함되어 있는지 확인합니다.
4. 문서가 최신이고 정확하며 현재 청구되는 실행 수수료와 일치하는지 확인합니다.
5. 문서에 제공된 정보를 사용하여 독립적으로 실행 수수료를 계산하려고 시도합니다.
6. 계산된 실행 수수료가 현재 청구되는 실행 수수료와 일치하는지 확인합니다.

<a name="r19"> **[R19]** </a>
트랜잭션 발신자, 트랜잭션 전송자 및 개발자는 L2 트랜잭션이 제출되기 전에 예상 트랜잭션 수수료와 그 구성 요소를 볼 수 있어야 합니다(MUST).

[[R19]](#r19) 테스트 가능성:

전제 조건:

* L2 인스턴스가 설정되고 실행 중입니다.
* 트랜잭션 발신자, 트랜잭션 전송자 및 개발자가 L2에 액세스하고 상호 작용할 수 있습니다.
* L2는 현재 운영 조건 및 기타 관련 요인에 따라 트랜잭션 수수료를 추정하는 기능을 제공합니다.

테스트 단계:

1. 트랜잭션 발신자, 트랜잭션 전송자 및 개발자가 L2 인터페이스로 이동하여 새 트랜잭션 생성 프로세스를 시작합니다.
2. L2 인터페이스가 트랜잭션 발신자, 트랜잭션 전송자 및 개발자에게 예상 트랜잭션 수수료와 그 구성 요소(실행 수수료, 데이터 수수료 및 우선순위 수수료)를 표시합니다.
3. 트랜잭션 발신자, 트랜잭션 전송자 및 개발자가 L2에 트랜잭션을 제출합니다.

테스트 통과 기준:

* L2 인터페이스가 예상 트랜잭션 수수료와 그 구성 요소를 정확하게 표시합니다.
* 트랜잭션 발신자, 트랜잭션 전송자 및 개발자가 트랜잭션을 제출하기 전에 예상 트랜잭션 수수료와 그 구성 요소를 검토하고 확인할 수 있습니다.

<a name="r20"> **[R20]** </a>

* 트랜잭션 수수료와 그 구성 요소가 L2에서 트랜잭션이 처리되고 최종화된 후에도 존재하고 정확합니다.

<a name="r30"> **[R30]** </a>
모든 L2 블록은 L2 프로토콜의 사양에 따라 해당 L2 블록의 누적 트랜잭션 수수료가 전체 또는 비율로 전송되는 하나 이상의 L2 계정 주소를 포함해야 합니다(MUST).

[[R30]](#r30) 테스트 가능성:

전제 조건:

* L2 프로토콜과 그 사양이 구현되고 테스트되었습니다.
* L2 인스턴스가 설정되어 기능합니다.
* L2 트랜잭션 세트가 생성되어 L2에서 최종화되었습니다.

테스트 단계:

1. 모든 L2 블록에 누적 트랜잭션 수수료를 받을 하나 이상의 L2 계정 주소가 지정되어 있는지 확인합니다.
2. 블록의 각 L2 트랜잭션에서 누적된 트랜잭션 수수료가 지정된 L2 계정 주소로 전송되었는지 확인합니다.
3. 누적된 트랜잭션 수수료의 전송이 L2 블록에서 누적 트랜잭션 수수료를 받도록 지정된 계정 주소에 전체 또는 비율로 수행되었는지 확인합니다.

테스트 통과 기준:

* 테스트 L2 트랜잭션을 포함하는 L2에서 최종화된 모든 L2 블록에 대해 1, 2, 3단계가 통과해야 합니다.

<a name="r31"> **[R31]** </a>
하나 이상의 L2 트랜잭션이 L2의 레이어 1에서 최종화되기 전에 되돌려지면, 영향을 받는 트랜잭션의 트랜잭션 수수료는 트랜잭션 전송자에게 청구되어서는 안 됩니다(MUST NOT).

참고로, L2 트랜잭션이 최종화되기 전에 되돌려진 이유는 중요하지 않습니다.

[[R31]](#r31) 테스트 가능성:

전제 조건:

* L2 인스턴스가 설정되어 기능합니다.
* L2의 레이어 1에서 최종화되지 않은 유효한 L2 트랜잭션이 적어도 두 개 있습니다.
* 각 L2 트랜잭션에는 유효한 트랜잭션 전송자가 있습니다.
* L2 트랜잭션에는 관련된 트랜잭션 수수료가 있습니다.
* L2 프로토콜은 레이어 1에서 최종화되기 전에 트랜잭션을 되돌리는 것을 지원합니다.

테스트 단계:

1. 트랜잭션 전송자의 초기 잔액이 기록되었는지 확인합니다.
2. L2 트랜잭션을 L2에 제출합니다.
3. L2 트랜잭션이 유효하고 트랜잭션 수수료가 올바르게 계산되었는지 확인합니다.
4. L2의 레이어 1에서 최종화되기 전에 L2 트랜잭션 중 하나를 되돌리려고 시도합니다.
5. 되돌려진 L2 트랜잭션의 트랜잭션 수수료가 트랜잭션 전송자에게 청구되지 않았는지 확인합니다.
6. L2의 레이어 1에서 나머지 L2 트랜잭션을 최종화합니다.
7. 최종화된 L2 트랜잭션에 대해 트랜잭션 수수료가 트랜잭션 전송자에게 청구되었는지 확인합니다.
8. 트랜잭션 전송자의 잔액이 그에 따라 업데이트되었는지 확인합니다.

테스트 통과 기준:

* L2 트랜잭션이 최종화된 후 트랜잭션 전송자의 잔액이 올바르게 업데이트됩니다.
* 최종화된 L2 트랜잭션의 트랜잭션 수수료가 해당 트랜잭션 전송자에게 청구됩니다.
* 되돌려진 L2 트랜잭션의 트랜잭션 수수료가 해당 트랜잭션 전송자에게 청구되지 않습니다.

<a name="d1"> **[D1]** </a>
L2 메타 트랜잭션이 처리 L2의 레이어 1에서 최종화되기 전에 되돌려지고 트랜잭션 발신자가 중개자에게 트랜잭션 수수료를 청구받은 경우, 되돌려진 메타 트랜잭션의 트랜잭션 수수료는 중개자가 트랜잭션 발신자에게 환불해야 합니다(SHOULD).

[[D1]](#d1) 테스트 가능성:

전제 조건:

* L2 인스턴스가 설정되어 테스트에 사용 가능합니다.
* L2 메타 트랜잭션 세트가 트랜잭션 전송자에 의해 제출되고 L2에서 처리되었습니다.
* 일부 L2 메타 트랜잭션이 L2의 레이어 1에서 최종화되기 전에 되돌려졌습니다.

테스트 단계:

1. L2에 L2 메타 트랜잭션 세트를 제출합니다.
2. 트랜잭션 전송자와 개발자가 L2 메타 트랜잭션을 제출하기 전에 예상 트랜잭션 수수료와 그 구성 요소를 볼 수 있는지 확인합니다.
3. 중개자가 L2 메타 트랜잭션이 L2에서 처리된 후 트랜잭션 수수료와 그 구성 요소를 보고 기록할 수 있는지 확인합니다.
5. L2의 레이어 1에서 최종화되기 전에 일부 L2 메타 트랜잭션을 되돌립니다.
6. 중개자가 되돌려진 메타 트랜잭션에 대해 기록된 트랜잭션 수수료를 트랜잭션 전송자 계정 또는 중개자 계정에서 트랜잭션 발신자 계정으로 반환합니다.
7. 트랜잭션 발신자가 트랜잭션 수수료 환불을 받았고 올바른 값인지 확인합니다.

테스트 통과 기준:

* 중개자, 트랜잭션 전송자 및 개발자가 L2 메타 트랜잭션을 제출하기 전에 예상 트랜잭션 수수료와 그 구성 요소를 볼 수 있습니다.
* 중개자가 L2 메타 트랜잭션이 L2에서 처리된 후 트랜잭션 수수료와 그 구성 요소를 볼 수 있습니다.
* 중개자가 L2 메타 트랜잭션이 되돌려질 때 올바른 양의 트랜잭션 수수료를 환불로 트랜잭션 발신자에게 반환합니다.
